## What this audits

"Byte-identical selection" (design D2) is only well-defined relative to a
registration sequence, and `entry_points()` order is venv-dependent. The committed
Mode-A oracle therefore fixes a **canonical** registration order (contracts sorted
by name). This audit re-runs the full probe corpus under a **permuted** registration
order (seed 1) and enumerates every probe whose equivalence tuple changes.

A tuple that changes under permutation is a **real multi-match tie** — a topic where
more than one dispatcher matches and the engine's fan-out order is the `_routes`
dict insertion order (`message_dispatch_engine.py:1845`). At cutover, the seam
consumes the same discovery output in the same process, so order variance hits both
implementations identically — which is why this is NOT a Mode-A parity failure. But
each tie is a place where production fan-out ORDER is discovery-order-dependent, a
latent hazard the seam work should make deterministic.

## Why report-only at S0

- These are not selection-set changes (the SAME dispatchers fire); only fan-out
  ORDER differs. Mode A pins the canonical-order tuple, and the seam consumes the
  same discovery output, so cutover parity is unaffected.
- Fixing them (deterministic fan-out ordering independent of discovery order)
  changes runtime behavior and belongs BEHIND the gate, after S3, under OMN-12525 —
  not in this parity PR (design D1: the gate is not the place to fix routing).

## Follow-up

Each order-sensitive topic is a candidate for a deterministic fan-out ordering fix
(e.g. sort selected dispatchers by dispatcher_id before execution) tracked under
OMN-12525. Re-run this audit after any change to route registration or fan-out
ordering; the count should trend to zero as ties are made deterministic.
